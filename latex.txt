\begin{abstract}
Large language models frequently produce syntactically invalid or schema-violating outputs when asked to generate structured function calls. I investigate whether QLoRA fine-tuning on GPT-2 can improve JSON validity, schema adherence, and exact-match accuracy for single-turn tool calling compared to a prompt-only baseline. After building a complete pipeline---dataset construction, prompt formatting, QLoRA training, inference, and a three-metric evaluation harness---I find that QLoRA fine-tuning on 255 training examples across 10 tool types raises JSON validity from 0\% to 75.6\%, schema adherence from 0\% to 57.8\%, and exact match from 0\% to 8.9\%, demonstrating that even a small amount of structured fine-tuning can teach GPT-2 to produce well-formed tool calls the majority of the time.
\end{abstract}

\section{Approach}

My project extends the default GPT-2 sentiment classification task to structured tool-call generation, inspired by Gorilla \cite{patil2023gorilla}. The core question is whether parameter-efficient fine-tuning can teach a small language model to reliably emit valid, schema-conforming JSON function calls.

\paragraph{Task formulation.}
Given a natural-language instruction and a tool specification (function name, parameter schema with types and required fields), the model must output a single JSON object of the form \texttt{\{"name": ..., "arguments": \{...\}\}}. I evaluate on single-call settings across 10 diverse tool types (weather lookup, flight booking, email composition, reminders, restaurant search, translation, alarms, stock prices, currency conversion, and tip calculation).

\paragraph{Prompt design.}
Each input is formatted with a standardized template that includes the tool schema as a JSON string and an explicit instruction to return only a JSON object:

\begin{verbatim}
You are given a tool specification and a user
instruction. Return a single JSON object with
exactly these keys: "name", "arguments".
Do not output any other text.

TOOL_SPEC: {<tool schema JSON>}
INSTRUCTION: {<user query>}
\end{verbatim}

\paragraph{QLoRA fine-tuning.}
I use QLoRA \cite{dettmers2023qlora}: the base GPT-2 (124M parameters) is loaded in 4-bit precision via \texttt{bitsandbytes}, and low-rank adapters (rank $r{=}16$, $\alpha{=}32$, dropout $0.05$) are applied to the attention projection matrices using HuggingFace PEFT \cite{hu2022lora}. Training uses the HuggingFace \texttt{Trainer} with causal language modeling loss over the full prompt--completion sequence.

\paragraph{Baselines.}
My primary baseline is \textbf{prompt-only GPT-2}: the same base model with identical prompts but no weight updates. I plan to add a non-quantized LoRA baseline and an optional constrained-decoding baseline in future work.

\paragraph{Implementation.}
I wrote the data pipeline (\texttt{build\_dataset.py}, \texttt{split\_dataset.py}, \texttt{format\_prompts.py}), training data generator (\texttt{generate\_training\_data.py}), training script (\texttt{train\_qlora.py}), inference engine (\texttt{infer.py}), evaluation harness (\texttt{eval\_json.py}, \texttt{eval\_schema.py}, \texttt{eval\_em.py}), and stress-test generator (\texttt{stress\_tests.py}). Model loading, tokenization, and QLoRA adapter application rely on \texttt{transformers} \cite{wolf2020transformers} and \texttt{peft}.

\section{Experiments}

\paragraph{Data.}
I programmatically generated 300 tool-call examples spanning 10 tools, each with 6--10 natural-language template phrasings and randomized slot values (cities, dates, amounts, etc.). Each example contains an instruction, a full JSON Schema tool specification, and a gold-standard target call. After filtering for unambiguous examples, I split 85/15 into 255 training and 45 dev examples. The 10 tools cover diverse argument types (strings, numbers, integers, enums) and arity (1--3 required arguments, 0--1 optional).

\paragraph{Evaluation method.}
I report three complementary metrics on the held-out dev set:
\begin{itemize}
    \item \textbf{JSON validity rate}: fraction of outputs that parse as valid JSON.
    \item \textbf{Schema adherence rate}: fraction whose keys, types, and required arguments match the provided tool schema (validated programmatically against the schema, not just the gold target).
    \item \textbf{Exact match (EM)}: fraction where function name and canonicalized arguments exactly match the gold target call.
\end{itemize}

\paragraph{Experimental details.}
I ran two rounds of experiments. The first used 14 training examples (toy dataset, rank $r{=}8$, $\alpha{=}16$, lr $5{\times}10^{-5}$, 3 epochs). The second---reported here---used 255 training examples with rank $r{=}16$, $\alpha{=}32$, learning rate $3{\times}10^{-4}$, batch size 4, max sequence length 384, and 5 epochs (320 optimizer steps, completing in ${\sim}30$ seconds on a single GPU). Training loss decreased from 6.58 (toy) to 0.75 (300-example run). Inference used greedy decoding (\texttt{do\_sample=False}) with \texttt{max\_new\_tokens=128}. All experiments used seed 11711.

        QLoRA GPT-2 (toy, 14 train)  & 0.0\%  & 0.0\%  & 0.0\% \\
        QLoRA GPT-2 (255 train) & \textbf{75.6\%} & \textbf{57.8\%} & \textbf{8.9\%} \\
        \bottomrule
    \end{tabular}
    \vspace{0.5em}
    \caption{Dev set results ($n{=}45$). QLoRA fine-tuning on 255 examples yields large gains in JSON validity and schema adherence over both the prompt-only baseline and the initial toy-data run.}
    \label{tab:results}

The prompt-only baseline produces 0\% valid JSON across all metrics---outputs are conversational continuations or echoed instructions. The toy QLoRA run (14 examples) also scores 0\%, though qualitative inspection shows early signs of JSON-adjacent structure. Scaling to 255 examples produces a dramatic improvement: 75.6\% of outputs parse as valid JSON and 57.8\% fully conform to the tool schema.

The error taxonomy for the 255-example model is informative: of the 19 failures, 11 are \texttt{invalid\_json} (unparseable output), 6 are \texttt{missing\_required\_arg} (valid JSON but omitting a required parameter), 1 is \texttt{missing\_key} (missing top-level \texttt{name} or \texttt{arguments}), and 1 is \texttt{unknown\_argument} (hallucinated parameter name). The gap between JSON validity (75.6\%) and exact match (8.9\%) suggests the model has learned the structural format but struggles with precise argument extraction from natural language, which is expected given GPT-2's limited capacity.

\section{Future Work}

I plan four improvements before the final report:

\begin{enumerate}
    \item \textbf{Completion-only loss masking}: modify the training loop so loss is computed only on target JSON tokens (not the prompt), focusing learning on structured output generation---the highest-impact single change.
    \item \textbf{Further data scaling}: increase to 500--1000 examples and incorporate real-world tool-call datasets (e.g., ToolBench \cite{qin2023toollm}).
    \item \textbf{Constrained decoding}: implement a JSON-grammar-constrained decoder or post-hoc JSON repair pass as an additional baseline.
    \item \textbf{Robustness evaluation}: run the existing stress-test suite (parameter renaming, field reordering, unseen tools, underspecified instructions) to measure generalization under distribution shift.
\end{enumerate}

I also plan to compare QLoRA against standard (non-quantized) LoRA fine-tuning to isolate the effect of 4-bit quantization on structured output quality.
